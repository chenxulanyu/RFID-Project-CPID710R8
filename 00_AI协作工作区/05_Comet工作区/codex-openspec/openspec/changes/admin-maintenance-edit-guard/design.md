## Context

后台维护页之前被改成了项目与任务统一保存，但现在用户希望恢复更清晰的分区式维护体验。与此同时，项目基础信息应默认锁定，避免管理员在新增任务时误改项目总信息。

## Goals / Non-Goals

**Goals:**
- 恢复项目区和任务区的分区布局。
- 让项目基础信息默认不可编辑，勾选确认后才解锁。
- 让项目保存与任务保存回到独立入口。
- 保持任务新增、更新、归档、恢复和错误提示可用。

**Non-Goals:**
- 不改变 CloudBase 持久化模型。
- 不移除任务列表筛选、归档和恢复能力。
- 不增加新的后端服务或权限系统。

## Decisions

### 1. 用显式确认解锁项目元数据
项目总信息默认禁用，只有用户显式勾选确认后才开放编辑。这样可以把“看”和“改”分开，减少误操作。  
Alternatives considered:
- 直接可编辑：误改风险太高。
- 用二次弹窗确认每次修改：打断感太强，不适合频繁维护。

### 2. 恢复分区式保存入口
项目基础信息与任务详情分开保存，按钮位置与其所属区块一致。这样操作路径更符合当前页面结构，也更便于定位失败来源。  
Alternatives considered:
- 保留单按钮统一保存：与用户明确反馈冲突。
- 自动保存：会增加误触和请求频率，不利于维护场景。

### 3. 继续沿用现有 repository 校验与回读
保存失败的根因优先从现有 CloudBase 回读校验和表单校验链路排查，不额外引入新的存储逻辑。  
Alternatives considered:
- 重写保存流程：成本高且风险大。
- 直接忽略失败提示：会重新制造“看似成功”的假象。

## Risks / Trade-offs

- [Risk] 分区保存后，用户可能忘记保存其中一块。→ Mitigation: 按区块明确按钮与提示，并保留清晰错误反馈。
- [Risk] 项目信息锁定后，部分用户会多一步确认。→ Mitigation: 默认安全优先，确认只需一次点击。
- [Risk] 保存失败若来自 CloudBase，界面调整无法单独解决。→ Mitigation: 将失败提示和回读结果保留在同一 change 中验证。

## Migration Plan

1. 恢复后台页面布局为项目区 + 任务区。
2. 为项目区增加锁定/解锁状态。
3. 分拆保存入口并补充测试。
4. 验证保存失败、解锁编辑和任务新增流程。

Rollback:
- 如分区布局导致维护效率下降，可恢复统一按钮，但保留项目锁定逻辑。

## Open Questions

- 项目区解锁后是否需要自动在保存后重新锁定。
- 保存失败提示是否需要进一步区分“项目失败”和“任务失败”。
