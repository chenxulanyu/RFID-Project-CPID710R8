## Context

当前线上 CloudBase 项目文档只包含部分字段，任务集合里也存在旧格式或不完整记录。前端在读取缺失周期字段时会直接显示 `undefined`，后台在写入失败时又没有检查 CloudBase 返回结果，因此会出现“保存成功但其他设备看不到变化”的假象。与此同时，用户希望后台编辑项目基础信息和任务详情时能够一次性保存。

## Goals / Non-Goals

**Goals:**
- 让 CloudBase 写入具备可验证性，失败时明确返回错误。
- 让 CloudBase 读取在项目字段缺失时回退到种子默认值，避免 `undefined`。
- 让后台通过单一保存动作同时提交当前项目和任务编辑内容。
- 保留任务归档与恢复为独立动作。

**Non-Goals:**
- 不修改扣子部署配置文件。
- 不改变任务归档、恢复、列表筛选的既有交互。
- 不引入新的外部存储服务或新的后台 API。

## Decisions

### 1. CloudBase 写入采用“返回码 + 回读校验”
CloudBase Web SDK 的 `set()` 可能返回错误对象而不直接抛异常。为避免假成功，repository 在写入后先检查返回体，再立刻回读比对核心字段。

Alternatives considered:
- 仅依赖 `set()` 的 promise reject: 风险是错误对象被当作成功值。
- 只看回读结果不看返回体: 风险是 API 返回错误却未立即发现。

### 2. 项目读取对缺失字段使用种子回退
项目文档缺少 `plannedStartDate` / `plannedEndDate` 时，直接渲染会导致 dashboard 和 timeline 异常。设计上采用种子默认值回退，确保展示层稳定。

Alternatives considered:
- 直接报错并阻断页面: 对线上容错较差。
- 在 UI 层单独做空值处理: 容易扩散到多个展示组件。

### 3. 后台统一保存当前项目和当前任务
项目基础信息和任务详情在用户心理上属于同一编辑上下文，因此后台提供一个保存动作更符合使用场景。实现上保存项目后再保存任务，最后统一刷新读取结果。

Alternatives considered:
- 保留两个按钮: 与用户预期不一致。
- 自动实时保存: 误操作风险高，且会增加请求频率。

## Risks / Trade-offs

- [风险] 回读校验会增加一次 CloudBase 读取。→ [缓解] 仅在保存动作发生时执行，频率低，换取更可靠的成功判断。
- [风险] 种子回退会掩盖部分 CloudBase 数据缺失。→ [缓解] 仅对项目必要字段做回退，同时保留原始远端文档作为数据源。
- [风险] 单按钮保存会让项目和任务的错误一起暴露。→ [缓解] 错误信息直接返回给用户，避免出现半成功状态。

## Migration Plan

1. 更新 repository 读取和写入逻辑。
2. 更新后台保存交互为统一按钮。
3. 运行测试和构建验证。
4. 如线上仍有旧格式数据，依赖新回退逻辑和回读校验继续稳定运行。

Rollback:
- 如统一保存带来交互问题，可恢复双按钮布局，但保留 CloudBase 回读校验。
- 如 CloudBase 回读校验导致不必要的失败，可先降级为仅检查返回码，但不建议作为默认长期方案。

## Open Questions
