## Context

后台维护页当前已经支持项目元数据和任务信息分区，但项目日期修改时仍可能触发 CloudBase 回读不一致误报；同时，任务列表、任务操作区和前台甘特图在密度与对齐上还有明显可读性问题，需要一次性收拢修正。

## Goals / Non-Goals

**Goals:**
- 让项目元数据保存的回读确认更稳健，避免正常修改日期时误报失败。
- 让后台任务列表充分利用页面高度，减少无意义的短列表滚动。
- 让任务保存按钮与归档/恢复按钮在同一区域内，降低操作跳转。
- 让前端任务主文案和甘特图标签更符合阅读顺序。
- 让甘特图的开始/结束时间、百分比和 100% 标签都不溢出。

**Non-Goals:**
- 不改 CloudBase 数据模型。
- 不新增新的保存模式或权限系统。
- 不改变项目任务的业务字段定义。

## Decisions

### 1. 项目元数据保存使用“写后回读 + 归一化 + 轻量重试”
项目保存继续以回读结果作为确认，但回读比较不直接做原始对象全等，而是把关心的字段先归一化，再做字段级比对；如果首次回读不一致，允许一次短暂重读后再判定。这样可以降低 CloudBase 在时间字段或字段序列化上的轻微差异导致的误报。

### 2. 左侧任务列表按视口高度拉伸
任务列表容器采用更稳定的纵向尺寸约束，让列表区域随页面剩余高度拉伸，而不是只按内容自然高度收缩。这样在任务较少时可以减少空白浪费，在任务较多时仍保留正常滚动。

### 3. 任务保存入口与归档入口并列
任务保存按钮移动到归档/恢复按钮旁边，放在同一个操作行里。这样一个任务的维护动作会聚在一起，减少“上面保存、下面归档”的视觉割裂。

### 4. 前台文案强调顺序统一
任务详情中优先强调项目内容，任务名称维持常规字重。这样和业务阅读习惯一致，也能和表格、时间轴中的分组信息保持一致。

### 5. 甘特图条内双端标签
时间轴任务条内部采用左右双端布局：开始时间贴左，结束时间贴右，百分比独立占据条内可用空间。100% 标签使用防压缩、防截断策略，避免在短条或高完成度条中被挤掉。

## Risks / Trade-offs

- [Risk] 写后回读加重试会让保存反馈稍慢。→ Mitigation: 仅重试一次，且只在不一致时触发。
- [Risk] 任务列表拉伸后，极短窗口下会更依赖滚动。→ Mitigation: 保持最小高度与溢出滚动，不强行压缩内容。
- [Risk] 甘特图双端标签可能挤压进度显示。→ Mitigation: 给百分比与时间标签分配明确的布局槽位，并在窄条上优先保留可读性。

## Migration Plan

1. 先稳定 CloudBase 项目保存校验。
2. 再调整后台布局和按钮位置。
3. 最后统一前台文案与甘特图标签排版。
4. 补充回归测试后完成验证。

Rollback:
- 如回读重试带来不可接受延迟，可退回单次回读，但保留归一化比对。
- 如甘特图双端标签在极端窄屏下过密，可退回到“条内标签 + tooltip”降级方案。
